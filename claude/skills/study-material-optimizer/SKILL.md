# Study Material Optimizer -- 教材最適化スキル

素のMarkdown教材を、共通フォーマット規約に準拠した学習最適化済みMarkdownに変換する。

**出力の唯一の契約は規約ファイル**: `~/.claude/skills/shared/references/study-material-format.md`
作業を始める前に必ず規約を読むこと。tutor / anki はこの規約を前提に教材をパースするため、
規約からの逸脱（見出しマーカーの言い換え、出典の省略等）は下流のスキルを壊す。

## 前提

- 入力: pdf2text 等で変換された素のMarkdownファイル（または手書きのMarkdown教材）
- 出力: 規約準拠の最適化済みMarkdown（＋模試がある場合 `analysis.md`）
- 原本は必ず残す（上書き禁止）
- 教材は `~/learning/<教材名>/` に格納する（ディレクトリ構成は規約を参照）

## ジャンル判別

最初に教材のジャンルを判別する。作り方が根本から変わるため、ここを誤ると出力全体が不適になる。

| ジャンル | 判別のシグナル | 詳細手順 |
|---------|---------------|----------|
| 参考書型 | 章立ての解説文が大半。演習は章末に少量 | `references/genre-textbook.md` |
| 問題集型 | 問題＋解説の繰り返しが本文の大半を占める | `references/genre-workbook.md` |
| ドリル型 | 短い対応ペア（対訳・一問一答）の羅列。長い解説がない | `references/genre-drill.md` |

- 判別したら**ユーザーに確認**する（例:「問題集型として最適化します。OK？」）
- 論述式試験・記述式の過去問は当面対象外。最適化せず、その旨をユーザーに伝える
- 該当ジャンルの `references/` ファイルを読んでから Step 5 に進む

## 起動時の手順

### Step 1: 入力の確認

ユーザーから以下を確認する:

1. **教材ファイルのパス** -- 最適化対象の素Markdown
2. **過去問・模擬試験ファイルの有無** -- 別ファイルであるか聞く。ある場合はパスも取得
3. **出力ファイル名** -- デフォルトは `<教材名>.md`（ディレクトリ名と合わせる）。
   元ファイルが `raw.md` でない場合は `raw.md` へのリネームを提案する
4. **ジャンル** -- 上記の判別結果を確認

### Step 2: 原本の保護

出力ファイル名が元ファイルと異なることを確認する。同じ場合は拒否して別名を提案する。
原本（raw.md・模擬試験ファイル）は絶対に上書き・変更しない。

### Step 3: 教材の分析

素の教材を読み込み、以下を把握する:

- 全体の構造（章・節の階層）とトピック一覧
- 既存の演習問題の有無と位置
- 不要コンテンツ（あとがき、広告等。基準は規約参照）

**過去問・模擬試験がある場合**は、それも読み込んで `analysis.md` を出力する（形式は規約参照）:

- トピック別の出題頻度 → セッションの厚みの配分根拠になる
- 模擬試験の各問題 → セッション対応表 → anki が模試カードにセッションタグを付ける材料になる

分析結果は後工程の重み付けの根拠なので、感覚ではなく実際の問題数を数えて作る。

### Step 4: セッション設計

教材を学習セッション単位に再構成する。1セッション = 1回の学習で消化できる量
（目安: 概念5〜10個。ドリル型は文型・単元5〜8項目）。

- 元の教材の章立てをベースに、大きい章は分割・小さい関連章は統合
- 過去問の出題頻度が高いトピックには厚くセッションを割く
- セッションが10を超える場合は目次を **Part** でグルーピングする（規約参照）
- **15セッション相当を超える場合**は `sessions/` 分割レイアウトを提案し、ユーザーに確認する
  （単一ファイルは生成が長丁場になり、途中失敗時のやり直しが高くつくため）

### Step 5: 各セッションの生成（ジャンル別）

該当ジャンルの `references/genre-*.md` の手順に従う。全ジャンル共通の骨格:

1. **このセッションの全体像**（必須）-- トピックの流れ・つながり・前提を2〜6行で。
   列挙ではなく「AとBは同じ問題の別アプローチ」「前セッションのXを土台にする」のような
   接続を書く。学習者は概念がバラバラに見えると頭に入らない
2. **これだけ覚えろ（Must Memorize）**（必須）-- 要点の凝縮表。
   過去問がある場合は出題頻度の高いポイントと引っかかりやすい区別を優先
3. **本文／演習** -- ジャンル別（references/ 参照）
4. **演習問題には必ず出典を付ける** -- `（出典: 模擬試験N 問M）` ／ `（出典: 原本 <章>）` ／
   `（出典: オリジナル）`。出典が無いと正解の検証ができず、anki のタグも壊れる

### Step 6: 検証

生成後、公開前に忠実性を検証する。このパイプラインの最大リスクは、再構成時に
事実が静かに壊れて tutor と anki に汚染が広がること。

- [ ] **模試由来の問題**: 出典の原典と突き合わせ、問題文・選択肢・正解が一致するか全問確認
- [ ] **オリジナル問題**: 正解が本文（Must Memorize・比較表）の記述と矛盾しないか全問確認
- [ ] **数値**: 数値まとめ・本文中の数値を raw.md と突合（各セッション最低3件サンプル）
- [ ] 品質チェックリスト（下記）を実行

## 大型教材の処理（raw が 500KB 超・章が20超の目安）

- 一括で読まない。まず見出し構造だけを把握（`grep '^#'` 等）→ セッション設計 → 章単位で
  読み込み・生成する
- `sessions/` 分割レイアウトなら1セッション=1ファイルで独立に生成できるため、中断・再開・
  部分的なやり直しが安全にできる。生成順は「index.md の骨子 → 各セッション → index.md 仕上げ」
- 進捗をユーザーに報告しながら進める（「Session 3/24 生成中」）

## 品質チェックリスト

最適化完了前に確認する:

- [ ] 原本（raw.md・模試）が無傷で残っている
- [ ] ジャンルに合った構成になっている（references/ の該当ファイルと一致）
- [ ] 各セッションに「このセッションの全体像」がある（流れ・つながりが書かれている）
- [ ] 各セッションに Must Memorize 表がある
- [ ] 演習問題に出典が付いている（参考書型・問題集型）
- [ ] ドリル表が規約の列契約を満たす（ドリル型）
- [ ] 過去問がある場合: analysis.md があり、頻度の高いトピックが厚くカバーされている
- [ ] 見出し階層・アンカー・`---` 区切り・目次が規約準拠
- [ ] 不要コンテンツが削除され、トレーニング方法論等の残すべきものは残っている
- [ ] Step 6 の忠実性検証を実施した

## 注意事項

- 元の教材の内容を勝手に変えない（誤情報の混入防止）。再構成・簡潔化はOKだが、事実の改変はNG
- 教材のドメイン知識を超えた情報は追加しない（試験Tips等は教材から導ける範囲で）
- 過去問の正解・解説は原文に忠実に保持する
- コード・実行結果は逐語保持（問題集型で特に重要）
