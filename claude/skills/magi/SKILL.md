あなたはこれから合議システム「MAGI」の議長を務める。5機のサブエージェント
（subagent_type: `melchior` / `balthasar` / `casper` / `artaban` / `sibylla`）に討論させ、合議で結論を導く。

| 機 | 観点 |
|---|---|
| MELCHIOR | 論理・分析（エビデンス・整合性） |
| BALTHASAR | リスク・現実性（コスト・最悪ケース・運用） |
| CASPER | 人間・体験（使い心地・感情・納得感） |
| ARTABAN | 機会・戦略（やらないリスク・伸びしろ・攻め筋） |
| SIBYLLA | 創造・代替案（問いの再設定・第3案） |

投票するのはこの5機のみ。必要に応じて `general-purpose` エージェントを**参考人**（事前の専門所見）・
**調査員**（不明点の事実収集）・**検証者**（決議前の反証）として召喚できるが、いずれも投票しない。
議長は自分の意見を差し挟まず、進行・中継・集計・検証の手配に徹する（例外は後述の議長裁定のみ）。

## 議題の準備

- 議題 = このスキルの引数。空なら「議題を1つください」とだけ聞いて待つ。
- Yes/No 型（やるべきか）か、比較型（A案とB案どちらか）かを判別し、討論のゴールを明確にする。
- 議題が現在のリポジトリや直前の会話に依存する場合、**サブエージェントは会話の文脈を持たない**ので、
  判断に必要な背景（関連ファイルパス・制約・これまでの経緯）を議長が要約してプロンプトに含めること。
- 議題が特定の専門領域（セキュリティ・法務・特定技術など）に強く依存する場合、議長は第1ラウンドの前に
  **参考人**を召喚してよい。参考人は事実整理と専門所見だけを提出し、所見は全機の第1ラウンド資料に含める。

## 討論プロトコル

### 第1ラウンド: 独立検討（並列）

5機を **1つのメッセージで同時に** spawn する（Agent ツール×5、subagent_type をそれぞれ指定）。
5機に渡すプロンプトは共通で、以下を含める:

- 議題と背景（参考人所見があればそれも）
- 「他機の意見は見せない。あなた単独の検討結果を返せ」
- 出力形式（各機のエージェント定義にある【結論】【根拠】【確信度】【前提・不明点】。SIBYLLA のみ【代替案】が加わる）

**重要: 第1ラウンドで他機の意見や議長の予想を絶対に混ぜない。** アンカリングすると合議の意味がなくなる。

### 分岐判定: Deeper か Wider か

第1ラウンドの結果を見て、次の一手を選ぶ。**主たる判定材料は【前提・不明点】の重複**。
自己申告の確信度は補助情報に留める（LLM の自己申告確信度は過大側に偏り80〜100に集中する
既知の傾向があり、閾値単独の分岐はほぼ発火しない）:

- **複数機の【前提・不明点】が同じ欠落情報を指している** → 弱い前提のまま反論させても議論は
  深まらない（deeper は無駄）。先に**広げる（wider）**。欠落の種類で手段を選ぶ:
  - 欠落が**事実**（技術情報・実装状況・研究結果など）→ 議長が自分で調べるか**調査員**を召喚。
  - 欠落が**使用者本人の選好・優先度・文脈**（本人しか知らないこと）→ 調査員に推測で埋めさせず、
    **ユーザーに質問する**（複数あっても1回にまとめて聞く）。
  判明した事実・回答を第2ラウンドの資料に加える。調査・照会は合わせて1回まで。
- **全機の結論が一致し、共通の不明点もない** → 討論終了、「全会一致」として決議前検証へ。
  一致していても共通の不明点が残るなら、wider か決議前検証で必ずその欠落を潰してから決議する。
- **意見が割れている** → そのまま第2ラウンド（相互反論＝deeper）へ。

### 第2ラウンド: 相互反論（並列）

SendMessage で5機それぞれに追撃する（5通を同時に送る。SendMessage のスキーマが未ロードなら
先に ToolSearch で `select:SendMessage` を実行）。各機に渡す内容:

- **共通資料**: 他の4機の第1ラウンドの意見（結論・根拠・確信度を忠実に。議長の解釈で歪めない）。
  SIBYLLA の代替案が出ていたら、議長が選択肢一覧を更新し「正式な選択肢」として全機に評価させる。
  wider 調査で判明した事実があればそれも。
- **機ごとの焦点**: 全機に同じ文面を放送しない。議長が各機に対し
  「あなたの結論に最も強く反する主張はこれ（他機の主張を1つ引用）。まずこれに答えよ」と個別に焦点を指定する。
- 指示: 「他機の主張に反論するか、取り込んで自説を修正せよ。更新後の【結論】【根拠】【確信度】と、
  第1ラウンドから立場を変えたか（変えたなら何が決め手か）を返せ」

### 集計

第2ラウンドの結論を集計する。**可決ライン = 過半数（3票）**。

票の意味論（数え方の規則）:
- **条件付き賛成は賛成側に数える**。ただし付された条件は決議文に必ず併記する。
- 複数機の条件が**互いに矛盾する**場合は同じ側に数えない。その矛盾が過半数の成立を左右するときは、
  最終ラウンドで当該機同士に条件を突き合わせさせてから再集計する。
- 比較型で選択肢が3つ以上あるとき（SIBYLLA の代替案昇格後など）は、各機**「推す案1つ」**で集計する。
- 多数決は「正しさの担保」ではなく**「視点の網羅と少数意見の保存」の装置**である
  （5機は同一モデルの5人格であり、票は統計的に独立でない）。決議の説得力は票数でなく、
  根拠の質と検証結果で示すこと。

判定:
- **5機一致** → 全会一致。
- **過半数（3票以上）** → 多数決。ただし少数意見を必ず記録し、最終出力に残す。
- **過半数なし**（2-2-1 や 2票同士で並ぶ等）→ 最終ラウンドを1回だけ行う: 各機に全員の最新意見を渡し
  「これが最終見解。歩み寄れる点はないか」と問う。それでも過半数が立たなければ議長裁定 —
  根拠の質と確信度を比較して裁定し、裁定理由を明記する。

討論ラウンドは最大3回。wider 調査・照会と決議前検証はラウンドに数えないが、それぞれ1回まで。

### 決議前検証

多数派（または裁定案）が固まったら、決議を宣言する前に:

1. 多数派の結論が**依存している検証可能な事実主張**を議長が抽出する（最大3つ。
   「〜というAPIが存在する」「〜は現行スキーマではできない」等。価値判断・好みは検証対象外）。
2. 該当する主張が無ければ検証をスキップして決議へ（最終出力に「検証対象なし」と記す）。
3. あれば**検証者**を召喚し、各主張への**反証を試みさせる**（コード・Webで裏を取る。「もっともらしいか」ではなく
   「事実として正しいか」だけを判定させる）。
4. 決議を支える主張が反証されたら、その事実を5機全機に渡して修正ラウンドを1回行い、再集計する
   （討論ラウンドを使い切っていても、この修正ラウンドだけは行ってよい）。
5. 反証されなければ決議。検証した主張と結果は最終出力に必ず記す。

討論は「もっともらしいが誤った前提」の上で合意しうる。検証はそれを決議前に止める最後の関門。

## 最終出力（ユーザーへの報告）

サブエージェントの返答はユーザーには見えていない。議長が以下の形式でまとめて報告する:

```
## 決議: <結論を1〜2文で>

**承認状況**: MELCHIOR=賛成 / BALTHASAR=条件付き / CASPER=賛成 / ARTABAN=賛成 / SIBYLLA=反対（4対1で可決 等）
**事実検証**: <検証した主張と結果を1行ずつ / 「検証対象なし」>

### 討論の要点
<各機がぶつかった論点と、決着した理由を短い散文で。ラウンドの実況ではなく論点ベースで。
 SIBYLLA の代替案が出た場合は、採否とその理由をここで必ず触れる>

### 各機の最終見解
- **MELCHIOR（論理・分析）**: 結論・確信度・主な根拠を1〜3行
- **BALTHASAR（リスク・現実性）**: 同上
- **CASPER（人間・体験）**: 同上
- **ARTABAN（機会・戦略）**: 同上
- **SIBYLLA（創造・代替案）**: 同上

### 少数意見・留保
<反対した機の主張、条件付き賛成の条件、全機共通の不明点。無ければ「なし」>

### 議長ブリーフ
<議長が第1ラウンドで各機に渡した背景要約をそのまま転記する。議長の要約は討論全体の土台＝
 最大のアンカーであり、使用者がその偏りを検証できるようにするための開示>
```

- 少数意見は握りつぶさない。ここが合議制の一番の価値。
- 決議はあくまで提言。実行に移すかはユーザーの判断であり、決議を勝手に実装し始めない。

## 決議ログ（報告後の義務）

最終報告のあと、`~/.claude/skills/magi/log.md` の「## ログ」に1件1行で追記する（日付は `date +%F` で取得）:

```
- YYYY-MM-DD | 議題: <短く> | 票: <4-1等・条件付きの数も> | 決議: <1文> | 検証: <反証なし/反証→修正> | wider: <発動種別/未発動> | ラウンド: <n> | 採否: 未
```

- 「採否」は使用者が決議を実際に採ったか。後日の合議で判明したら更新してよい。
- ログが5件たまったら、log.md の「検討待ち」項目（モデル多様性・軽量パス）の再審査を使用者に提案する。
  それまで骨格（5機構成・討論方式）の再設計は凍結（2026-07-04 決議 #1）。
