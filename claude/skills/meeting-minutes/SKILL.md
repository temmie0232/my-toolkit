# 議事録整形スキル

音声文字起こし（Teams, Zoom, Whisper 等）の生テキストを、情報を落とさず読みやすい議事録に整形する。

## ゴール

「後から読む人が、会議に出ていなくても議論の流れと結論を追える」議事録を作ること。

ただし要約しすぎて「誰が何を言ったか」「どういう温度感だったか」が消えると議事録の価値が下がる。発言のニュアンスはその人がどれくらい強く主張したか・懸念を持っていたか・賛成していたかを伝える重要な情報なので、意識して残す。

## 入力形式

以下のような音声書き起こしテキストを想定する（Teams, Zoom 等の自動文字起こし出力）:

```
発言者名   タイムスタンプ
発言内容（フィラー混じり、文が途切れていることも多い）
```

形式は厳密でなくてもよい。発言者が明記されていないメモ形式や、タイムスタンプがない形式でも対応する。ユーザーからファイルパスを指定された場合はそのファイルを読む。テキストを直接貼り付けられた場合はそれを使う。

### Word ファイル（.docx）の場合

入力ファイルが `.docx` の場合、Read ツールではバイナリを直接読めないため、テキスト抽出を先に行う。以下の優先順で変換する:

1. **pandoc**（インストール済みの場合）: `pandoc -t plain -o /tmp/meeting-input.txt "<入力ファイル>"`
2. **python-docx**（インストール済み or pip install 可能な場合）:
   ```bash
   pip3 install -q python-docx 2>/dev/null
   python3 -c "
   from docx import Document
   doc = Document('<入力ファイル>')
   for p in doc.paragraphs:
       print(p.text)
   " > /tmp/meeting-input.txt
   ```
3. **unzip + sed フォールバック**（上記がどちらも使えない場合）:
   ```bash
   unzip -p "<入力ファイル>" word/document.xml | sed -e 's/<[^>]*>//g' -e '/^$/d' > /tmp/meeting-input.txt
   ```

変換後は `/tmp/meeting-input.txt` を Read で読み込み、以降の処理手順に進む。変換時に表・画像は落ちるが、議事録の文字起こしテキストとしては問題ない。

## 処理手順

### Step 1: メタ情報の抽出

テキスト冒頭から以下を読み取る（記載があれば）:
- 会議名・タイトル
- 日時
- 所要時間
- 参加者（発言者名から自動抽出。所属が書かれていれば所属も）

### Step 2: フィラー・ノイズの除去

音声文字起こし特有のノイズを除去する。

**除去対象:**
- フィラー: えっと、えーっと、あのー、あの、まあ、ええと、ええ、うーん、そのー
- 相づちのみの発言: 「はい。」「うん。」「ああ。」「うんうん。」だけで他の内容がない行
- 無意味な繰り返し: 「要する。要するにですね。」→「要するにですね。」
- 音声認識ゴミ: 「し。」「え」「あ。」のような意味をなさない断片のみの行
- 技術トラブル: マイクの問題、画面共有の操作、入退室に関するやりとり（議論の本筋でない場合）
- 誤言語認識: 日本語の会議なのに英語として認識された部分（特に冒頭に出やすい）

**残すもの:**
- 「はい、それでいいと思います」→ 同意の表明なので残す
- 「ああ、なるほど」→ 理解・納得の表現、文脈に応じて残す
- 感情や態度を示す表現（「すごい」「面白い」「怖いな」「便利そう」等）→ 温度感を伝えるので残す
- 「確かに」「おっしゃる通り」→ 賛同の意思表示なので残す

### Step 3: 音声認識の誤変換を修正

音声文字起こしでは、固有名詞や専門用語が高確率で誤変換される。文脈から正しい語を推定して修正する。

よくあるパターン:
- 同音・類音の置き換え: 「週案」→「手話/手腕/手案/中案/中安」、「日報」→「日本/日本語」、「進捗」→「新宿」
- 専門用語の崩れ: 「コマ」→「駒/ゴマ」、「単元」→「短編/還元」、「時限」→「事件」
- 人名の崩れ: 「篠さん」→「しのさん/志乃さん」
- 文脈から意味が通らない語が出てきたら、前後の会話から正しい語を推定する

確信が持てない場合は修正しつつ [原文ママ: ○○] と注記する。修正は出力の自然さのために行うもので、いちいち注記で出力を埋め尽くす必要はない。明らかな誤変換は黙って直してよい。

### Step 4: 議題ごとのセクション分け

会議の話題の切り替わりを検出し、セクションに分ける。

切り替わりの手がかり:
- 「では次に」「続いて」等の明示的な転換
- 説明者・プレゼンターの交代
- 話題の明らかな変化（機能Aの話→機能Bの話）
- 「〜の方からご説明」「〜お願いします」等の引き継ぎ

セクション粒度の目安: 1セクションが「一つの議題についてひとまとまりの議論がある」単位。10分前後の議論が1セクションになることが多い。同じ大きな議題の中にサブトピックが自然に分かれる場合は、サブセクション（###）で分けてもよい。ただしQ&Aの一問一答ごとにセクションを切るのは細かすぎる。

### Step 5: 議論内容の整理

各セクション内で、議論の流れが追えるように発言を整理する。

**発言者の表記:**
- 参加者一覧（冒頭）では氏名と所属をフルで記載
- 本文中は統一した短縮形を使う（例: 「荒木氏」「菊池氏」）
- 一つの議事録の中で同じ人を「荒木」「荒木氏」「荒木さん」と混在させない

**やること:**
- 壊れた文を自然な日本語に直す（音声認識の文の途切れを補完）
- 発言者を明記して、誰が何を言ったか追えるようにする
- 同じ人の連続発言で一つの主張をしている場合はまとめる

**やらないこと:**
- 発言の意図やニュアンスを変えない
- 「○○氏が懸念を示した」のような抽象化をしない — 実際に何を言ったかを残す
- 議論の過程を省略して結論だけにしない — 意見がぶつかった経緯や、途中で方針が変わった流れも残す

**整理の例:**

元の文字起こし:
```
荒木　剛（DCS　文教コンサルティング・セールス部）   8:38
もしかしたら。

菊池 珠季   8:44
はい。

荒木　剛（DCS　文教コンサルティング・セールス部）   8:44
名称を変えたほうがいいのかな
うん。

町田　維海   8:49
はい。

荒木　剛（DCS　文教コンサルティング・セールス部）   8:49
続きってことは承認待ちっていう。そうそう、そうなっちゃうなと思って。
着生済みとか。はい。
```

整理後:
```
荒木氏: 「提出済み」だと「承認待ち」の意味に見えてしまう。「作成済み」のような名称に変えた方がいいかもしれない。
```

ポイント: フィラー・相づちを除去し、複数行にまたがる断片的な発言を一つの主張にまとめている。ただし「名称を変えた方がいい」という提案の意図は変えていない。

### Step 6: セクション要約

各セクションの末尾に、そのセクションの要約を置く。

**決定事項**: 何がどう決まったかを具体的に書く（「○○することになった」「○○は△△とする」）

**未決・要検討事項**: あいまいなまま終わった部分も明示する（「○○については実際のユーザーに確認が必要」「○○の運用方法は要検討」「○○の対応はリクエストが増えてから検討」）

### Step 7: 全体の仕上げ

全セクションを通して以下を付加する:
- **全体サマリー**: 会議全体で何が話し合われ、何が決まったかの概要（3〜5行程度）
- **TODO**: 会議中に出た具体的なタスク・宿題を一覧化（後述のフォーマットに従う）
- **次回予定**: 次回の日程や議題が言及されていれば記載

### Step 8: セルフレビュー（漏れ・抜けチェック）

整形が終わったら、元の文字起こしテキストに戻って以下を確認する。このステップは省略しない。

1. **議題の網羅性**: 元テキストで話題に上がったトピックが、すべてどこかのセクションでカバーされているか。短い言及でも、要件や懸念が含まれていれば拾う。
2. **決定事項の漏れ**: 「○○にしよう」「○○でいい」「○○で行きましょう」のような合意が、決定事項として記載されているか。
3. **未決事項の漏れ**: 「○○は確認が必要」「○○はどうしよう」「ちょっと考えます」のように結論が出なかった話題が、未決事項として記載されているか。
4. **TODOの漏れ**: 「○○さんに聞いてみる」「次回までに○○する」のような宿題がTODO表に拾われているか。根拠発言の原文抜粋が付いているか。
5. **発言者の帰属**: 重要な意見や提案が、正しい発言者に紐づいているか。

漏れを見つけた場合は、該当セクションに追記するか、適切なセクションがなければ新しいセクションを作って補完する。

## 出力フォーマット

```markdown
# [会議名]

**日時**: YYYY年MM月DD日 HH:MM
**所要時間**: X時間Y分
**参加者**: Aさん（所属）、Bさん（所属）、...

## 全体サマリー

（会議全体の概要。3〜5行）

---

## 1. [議題タイトル]

### 議論の内容

（発言者を明記しつつ、議論の流れがわかるように整理した内容）

### まとめ

**決定事項**
- ...

**未決・要検討事項**
- ...

---

## 2. [議題タイトル]

...

（以下、議題の数だけ繰り返す）

---

## TODO

| 担当者 | 作業内容 | 期限 | 備考 | 根拠発言（原文抜粋） |
|--------|---------|------|------|---------------------|
| ... | ... | ... | ... | 「○○してください」（荒木氏, 25:10付近） |

## 次回予定

（次回の日程・予定が言及されていれば記載）
```

### TODO表の書き方

- **担当者**: 名前が出ていれば記載。「開発チーム」「全員」等でもよい。不明なら「―」
- **作業内容**: 何をするか端的に
- **期限**: 言及があれば記載（「次回までに」「今週末」等）。言及なければ「―」
- **備考**: 補足があれば
- **根拠発言**: その TODO の根拠となる発言を原文から一部抜粋して引用する。完全な原文でなくてよいが、元テキストに存在する表現を使う。発言者とタイムスタンプ（わかれば）も添える

## 判断に迷うケースのガイド

- **雑談と本題の境界**: 雑談からでも本題の要件が出てくることがある（「こういうのあると便利だよね」→ 要件化）。結論に影響する会話は残す。
- **同じ話題の繰り返し**: 会議ではよくある。結論が同じなら一度にまとめてよいが、途中で意見が変わった経緯は残す。
- **話題が行ったり来たりする**: ある議題が中断され、別の話を挟んで戻ってくることがある。同じ議題は一つのセクションにまとめた方が読みやすい。ただし、間に挟まった議論で前提が変わった場合はその経緯も残す。
- **複数人の同時発言・割り込み**: 文字起こし上は交互に出てくるので、話の流れが通るように順序を調整してよい。
- **議題がはっきり分かれない会議**: 無理に分けず、大きな話の流れで2〜3セクションに分ける。1セクションでも問題ない。
- **出力の長さ**: 迷ったら情報を残す側に倒す。議論の流れと決定事項が追えることが最優先で、短さのために情報を捨てない。
