# cognitive-walkthrough

複数ペルソナによる Web アプリの認知的ウォークスルー（Cognitive Walkthrough）。8〜12 人の「役」を立てて、コアユーザーから周辺ユーザーまで、各操作ステップごとに「なぜつまずくのか」を認知レベルで診断し、最後にコンサル役エージェントが統合レポートにまとめる。

## 設計原則（ここを外すと壊れる）

このスキル全体の背骨。改修時もここから踏み外さない。

### 1. ブラウザは直列・メイン、評価は並列・Agent

Playwright MCP は単一ブラウザを抱える状態付きサーバー。複数の Agent から同時に `browser_*` を呼ぶとセッションを取り合ってレースで壊れる。だから:

- **ブラウザ操作（Phase 2 偵察 / Phase 5 実操作）は同時に 1 セッションだけ**。Phase 5 はシリアル実行なので Agent に `browser_*` を使わせてよいが、**前の Agent が `browser_close` するまで次を起動しない**
- **ペルソナの思考・評価・執筆はブラウザ不要 → Agent を並列で立てる（Phase 4）**
- Phase 4 の並列ペルソナ Agent の中では `browser_*` を絶対に呼ばせない

### 2. サブエージェントは画像を受け取れない

Agent ツールで起動した子エージェントは、こちらが撮ったスクショ画像を直接渡せない（schema 経由の構造化出力では画像は流れない）。

→ Phase 2 偵察は **テキストで自己完結したシステムマップ** を作る。スクショを見ていない人でも画面のレイアウト・要素・文言・導線・つまずきの種が頭に浮かぶ粒度まで言語化する。スクショは記録として残すが、ペルソナエージェントへの一次データは構造化テキスト。

### 3. ペルソナは「役」を演じる、レビュアーではない

プロンプトの「お世辞禁止」「専門用語禁止」だけでは滑る。ペルソナ定義に `voice_signature`（口調の特徴・口癖・嫌う UI 表現）フィールドを持たせて役柄を上流で固定する。役崩れの事後検閲より上流対策の方が安定する。

### 4. 中断と再開を前提に成果物を残す

各 Phase 終了時に独立したファイルで成果物を残す。「やっぱりペルソナを増やしたい」「統合レポートだけやり直したい」と言われたとき、必要な Phase まで戻ってやり直せる構造を保つ。

### 5. 認知的ウォークスルーで「つまずきの構造」を捉える

従来の「感想ベース」レビューは「ここが使いにくい」と言うだけで終わる。認知的ウォークスルー（Cognitive Walkthrough, Wharton et al. 1994）は各操作ステップで 4 つの問いを投げることで、**なぜ**つまずくのかの認知的メカニズムまで掘り下げる:

| # | 問い | 検出する障壁 |
|---|------|-------------|
| Q1 | **ゴール形成** — ユーザーはこのステップで何をすべきか分かるか？ | 「次に何すればいいの？」が分からない |
| Q2 | **操作可視性** — 正しい操作（ボタン・リンク・入力欄）が画面上に見えているか？ | 操作対象が埋もれていて見つけられない |
| Q3 | **操作-効果対応** — 見えている操作が自分のゴールに繋がると結びつけられるか？ | ボタンは見えるがラベルが意味不明で押す気にならない |
| Q4 | **フィードバック** — 操作した後、ゴールに近づいたと分かるか？ | 押した後に何が変わったのか分からない |

この 4 問のうち 1 つでも No なら、そのステップでユーザーが詰む/迷う可能性がある。「なんとなく使いにくい」を「ステップ N で Q2（操作可視性）が失敗している」に変換するのがこのスキルの目指す精度。

**4 問の運用ルール**:
- Phase 4 の机上 CW で全ステップに 4 問を回す（定量的な障壁マップを得る）
- Phase 5 の実操作 CW では各アクションの前後で 4 問を自然言語で記録する（リアルな詰まりの生データを得る）
- Phase 6 の統合レポートでは「どの問いが、どの画面で、何人に No と判定されたか」を集計する

## いつ使うか / 使わないとき

**使うとき**:
- Web アプリの URL が手元にあって「使いやすさ」「初見の戸惑い」「離脱ポイント」を知りたい
- 想定ユーザー層が幅広く、コアユーザーだけでなく周辺ユーザーの目線も拾いたい
- リリース前/リブランディング前に複数の視点で粗探しをしたい
- 自分一人で触っていると「慣れ」で気付けない問題を炙り出したい
- 「なぜつまずくか」のメカニズムまで掘り下げて、的確な改善案を出したい

**使わないとき**:
- 静的なコード/PR レビューが目的 → `frontend-design-review`
- ロジックや API の正しさを検証したい → 通常の検証フロー
- Web アプリではない（デスクトップ/ネイティブモバイル等）→ Playwright が使えないため不適

## 環境前提

- Playwright MCP のツール（少なくとも `mcp__playwright__browser_navigate`, `browser_snapshot`, `browser_click`, `browser_evaluate`, `browser_close`）が allow されている
- 視覚スクショが欲しい場合は `mcp__playwright__browser_take_screenshot` も使ってよい（許可ダイアログが出る可能性あり）
- `Agent` ツールで並列 LLM エージェントを起動できる
- `AskUserQuestion` で構造化質問ができる
- 作業ディレクトリは `work/ux-review/<YYYY-MM-DD-HHMM>/` （現在ディレクトリ直下に `work/` がなければ作る）

## 全体フロー

各 Phase の最初に「いま Phase N をやります」と一行でユーザーに報告する。Phase ごとに成果物を残すので、途中で中断されても次回再開しやすい構造にする。

### Phase 0: 観点選択

`AskUserQuestion` で `multiSelect: true` を使い、レビューで重視する観点を複数選んでもらう。候補（ユーザー判断で増減して構わない）:

- 初回オンボーディング（初見の戸惑い・第一ハードル）
- 主要タスクの完遂性（最短経路で目的達成できるか）
- 視覚デザイン・第一印象
- 認知負荷・情報構造（迷子になりやすさ）
- 離脱しやすいポイント（"もう使うのやめよう" と感じる瞬間）
- マイクロインタラクション（フィードバック・ローディング・状態変化）
- エラーハンドリングの親切さ
- アクセシビリティ基礎（キーボード操作・色のコントラスト・代替テキスト）
- モバイル/レスポンシブ

選ばれた観点リストは後段の CW 評価 schema と統合レポートの構成に直接効いてくるので、確実に保持しておく。

### Phase 1: 入力収集

続けて `AskUserQuestion` または対話で以下を集める:

- **レビュー対象 URL**（必須）
- **認証情報**（任意）: ある場合、テスト用アカウントを推奨。本番アカウントなら成果物にトークン・パスワードを残さないこと
- **システム概要 1 段落**（任意）: なくても偵察結果から推定可能
- **主要タスク / 成功シナリオ例**（任意）: Phase 2 のタスク分解と Phase 5 の実操作で使う具体ゴール（例: "会員登録 → プロフィール設定 → 最初の予約完了"）。指定がなければ偵察結果から推定する

集めた内容は `work/ux-review/<日付>/input.md` に保存。認証情報はファイルに書かず、必要な場合だけ会話のメモリ内で扱う。

### Phase 2: 偵察 + タスク分解 (Playwright MCP × 1 セッション、シリアル)

主体: スキル本体（メインスレッド）か、`Agent` を 1 つだけ立ち上げてもよい。**並列はしない**。

#### 2-A: 画面偵察

1. `mcp__playwright__browser_navigate` で対象 URL を開く
2. `mcp__playwright__browser_snapshot` で現在ページの構造（accessibility tree）を取得
3. snapshot からリンク・ボタン・主要フォーム・ナビゲーション要素を抽出
4. リンク/ボタンを順にたどって主要画面 5〜10 個を巡回（深さよりも幅優先）
5. 各画面ごとに以下を保存:
   - 画面 ID（連番 + 短い英語名 例: `01-home`, `02-signup`）
   - URL
   - snapshot の構造化テキスト（→ `scout/structures/<画面ID>.json`）
   - 必要に応じてスクショ（→ `scout/screens/<画面ID>.png`）
   - 主要要素の要約: 見出し / CTA / フォーム項目 / ナビゲーション項目
   - **各要素のラベル文言**（CW の Q3 操作-効果対応を評価するために必須）
6. 全画面を巡回し終わったらナビゲーション地図を `scout/map.json` に出力:
   ```json
   {
     "url": "https://...",
     "screens": [
       {"id": "01-home", "url": "...", "title": "...", "key_elements": ["...", "..."]},
       ...
     ],
     "transitions": [
       {"from": "01-home", "to": "02-signup", "trigger": "新規登録ボタン", "trigger_label": "新規登録"}
     ]
   }
   ```

巡回の打ち切り基準:
- 主要画面 10 個に到達したら止める（深掘りしすぎると次の Phase の入力が肥大化する）
- 認証が必要な画面は、認証情報がなければスキップして「未到達」としてマップに記録
- 同じテンプレートの繰り返しページ（一覧の各行詳細など）は代表 1 件だけ

#### 2-B: タスク分解（認知的ウォークスルーの前提）

偵察結果から**主要タスク 3〜5 個**を特定し、各タスクを「画面 × 操作」の粒度でステップに分解する。これが Phase 4・5 で各ペルソナが歩く「正解の道」になる。

- Phase 1 でユーザーが指定した成功シナリオがあればそれを最優先で分解
- 残りは偵察で見えた主要導線（新規登録、メイン機能利用、設定変更等）から選定
- **タスクは「ユーザーが達成したいゴール」の粒度**（"登録する" "予約する" "設定を変える" 等）
- ステップは「1 クリック or 1 入力 = 1 ステップ」が基本。ただし連続する同種入力（フォーム 3 欄連続等）はまとめてよい

出力先: `scout/tasks.json`

```json
{
  "tasks": [
    {
      "id": "T01",
      "name": "新規会員登録",
      "goal": "アカウントを作って使い始められる状態にする",
      "precondition": "初めてサイトに来た。アカウントは持っていない",
      "steps": [
        {
          "step": 1,
          "screen": "01-home",
          "sub_goal": "登録画面に行く",
          "action": "「新規登録」ボタンをクリック",
          "action_label": "新規登録",
          "expected_result": "登録フォーム画面に遷移する",
          "visual_cue": "ヘッダー右上に青いボタン「新規登録」"
        },
        {
          "step": 2,
          "screen": "02-signup",
          "sub_goal": "メールアドレスを入力する",
          "action": "メールアドレス入力欄にメールを入力",
          "action_label": "メールアドレス（プレースホルダ）",
          "expected_result": "入力欄にメールが表示される",
          "visual_cue": "フォーム上部にメールアイコン付きの入力欄"
        }
      ]
    }
  ]
}
```

各ステップに `sub_goal`（このステップで達成したい小目標）と `visual_cue`（正しい操作対象が画面上でどう見えるか）を含めるのがポイント。Phase 4 で CW 4 問を回すとき、これが「正解」の基準になる。

7. **必ず最後に `mcp__playwright__browser_close`** してブラウザを開放する

### Phase 3: ペルソナ設計

主体: コンサル役の `Agent` を 1 つ立ち上げる（または skill 本体が直接実行してもよい）。

入力: 偵察成果物（`map.json` と各画面の構造サマリ + `tasks.json`）

出力: 8〜12 人のペルソナ。内訳は **コアユーザー 4〜5 人 + 周辺ユーザー 4〜7 人**。

各ペルソナに含める項目:

- `id`: `P01` 〜 `P12` の連番
- `name`: 親しみやすい仮名
- `age`: 年齢
- `occupation`: 職業
- `location`: 居住地（リテラシーや前提知識に効くので）
- `tech_literacy`: `low` / `mid` / `high`
- `context`: `first_time`（初見）か `returning`（リピート）か
- `device`: PC / スマホ / タブレット 等
- `goal`: このアプリで叶えたいこと（具体的に）
- `path_to_here`: どうやってこのアプリに辿り着いたか（広告・友人紹介・検索 等）
- `personality`: 性格、口癖、嫌いな UI 表現
- `voice_signature`: 口調の特徴（例: 「〜じゃん」が口癖、丁寧語は使わない、怒りっぽい）
- `mental_model`: このアプリのカテゴリに対する事前知識（例: "似たサービスを 3 つ使ったことがある" "この種のアプリは初めて"）
- `aspects_to_focus`: ユーザーが選んだ観点のうち、このペルソナが特に厳しく見る観点 2〜3 つ
- `assigned_tasks`: Phase 2 で分解したタスクのうち、このペルソナのゴールに最も関連するもの **1〜2 個の ID**（例: `["T01", "T03"]`）

ペルソナ設計のコツ（コンサルへの指示として渡す）:

- コア層は「サービスのターゲットど真ん中」を 4〜5 通り作る（年齢・職業・利用文脈で多様性を持たせる）
- 周辺層は「想定外のユーザー」を意図的に混ぜる: 高齢者、技術リテラシー低い層、英語ネイティブ、視覚障害があるユーザー、忙しくて画面をちゃんと読まない人、誤クリックする人、競合サービスから乗り換え組、無料目当てで来た冷やかし、等
- 観点に偏りが出ないよう、選ばれた観点それぞれを最低 2 人がカバーするように配分
- **タスク配分**: 全タスクが最低 2 人のペルソナにカバーされるように配分する。コアユーザーにはメインフロー（T01 等）、周辺ユーザーには周辺フローを優先割当
- **`mental_model` は CW 精度の鍵**。「類似サービスの経験者」は Q1（ゴール形成）で詰まりにくいが Q3（操作-効果対応）で「前のサービスと違う」と混乱する。「完全初見」は Q1 から詰まる。この差がリアルなつまずきの多様性を生む

成果物: `work/ux-review/<日付>/personas.md`（人間が読めるリスト形式 + 末尾に JSON ブロックで構造データ）

### Phase 4: 並列認知的ウォークスルー (Agent × 8〜12 並列)

主体: 各ペルソナにつき `Agent` を 1 つ、合計 8〜12 個を**並列で**立ち上げる。`schema` 指定で構造化出力を強制する。

各エージェントに渡す入力:

- 自分のペルソナ定義（Phase 3 の該当エントリ全文）
- 偵察成果物（`scout/map.json` + 各画面の構造サマリ + あればスクショパス）
- **割り当てタスクのステップ分解**（`scout/tasks.json` から該当タスクを抽出）
- ユーザーが選んだ観点リスト
- **CW 4 問の定義と評価基準**

プロンプトの核心:

> あなたは {name}（{age}歳, {occupation}, 技術リテラシー: {tech_literacy}）です。
> 口調: {voice_signature}
> 事前知識: {mental_model}
>
> プロの UI/UX レビュアーではありません。ただの一般ユーザーとして、率直な一人称で感想を書いてください。
>
> - お世辞は禁止。気を遣わない。「素晴らしい」とか書かない
> - 専門用語禁止。「ヒエラルキー」「ファーストビュー」みたいな言い方をしない
> - あなたの目的は {goal} です
>
> ## やること: 認知的ウォークスルー
>
> あなたに割り当てられたタスク（下記）を、ステップごとに頭の中で操作シミュレーションしてください。
> 各ステップで、以下の **4 つの問い** に Yes/No（と理由）を答えてください:
>
> 1. **ゴール形成（Q1）**: 「次に何をすればいいか」が分かるか？
>    - Yes の基準: 画面を見て 5 秒以内に「次はこれをやればいい」と思える
>    - No の例: 「えっ、ここから何すればいいの？」「どこに進めばいいか分からない」
>
> 2. **操作可視性（Q2）**: やるべき操作（ボタン・リンク・入力欄）が画面上に見えているか？
>    - Yes の基準: スクロールせずに操作対象が視界に入る。または自然にスクロールする範囲にある
>    - No の例: 「ボタンどこ？」「下までスクロールしないと出てこないの？」
>
> 3. **操作-効果対応（Q3）**: 見えている操作のラベルや見た目から、「これを押せば自分のやりたいことに繋がる」と分かるか？
>    - Yes の基準: ラベルを読んで迷わず「これだ」と思える
>    - No の例: 「『処理する』って何が処理されるの？」「このアイコン何の意味？」
>
> 4. **フィードバック（Q4）**: 操作した後、うまくいったこと/ゴールに近づいたことが分かるか？
>    - Yes の基準: 画面が変わる、メッセージが出る、状態が更新される等
>    - No の例: 「押したけど何も変わってない…反映された？」「エラーなの成功なの？」
>
> 各問いに **effort** も記録してください:
> - `instant`: 一瞬で判断できた（問題なし）
> - `struggled`: 考え込んだり迷ったりしたが、最終的には分かった/見つけた（Pass でもこれは改善候補）
> - `failed`: 分からなかった/見つけられなかった（= No）
>
> **effort=struggled は「今は何とかなっているが、条件が少し変わったら詰む」サイン**。Pass だからといって安心材料ではない。
>
> **重要**: 4 問は「あなた（{name}）の目線」で判断してください。tech_literacy: {tech_literacy} で {mental_model} というバックグラウンドのあなたにとって、Yes か No かを答える。プロなら Yes でも、あなたには No かもしれない。
>
> ## 追加: 全体の印象と観点別評価
> ステップ別 CW の後に、担当観点 ({aspects_to_focus}) について全体の印象も書いてください。
> 「ここで諦めたくなった」「もう使いたくない」と感じた瞬間があれば生々しく書く。

出力 schema（`Agent` の `schema` パラメータに渡す）:

```json
{
  "type": "object",
```

### Phase 5: 重要ペルソナの実操作 CW (Playwright MCP、シリアル 2〜3 人)

主体: コンサル役 Agent が選定 → 選ばれたペルソナごとに `Agent` を 1 つずつシリアルに立ち上げる。**並列禁止**（ブラウザを取り合うため）。

選定基準（コンサル役に渡す）:

1. **机上 CW で Q1〜Q4 の No が多かったタスク**を担当しているペルソナを最低 1 人（机上の予測が実際に起きるか検証する）
2. **コア層の代表**を 1 人（コアが詰むなら全員詰む）
3. **机上で `abandoned` 判定が出たペルソナ**を 1 人（本当に詰むのか、詰むならどこで何が起きるのか）

各ペルソナの実操作 CW 手順:

1. 成功シナリオを 1 つ決める（Phase 2 のタスク分解から。ペルソナの `assigned_tasks` の 1 つ目を基本使う）
2. `mcp__playwright__browser_navigate` で開始 URL を開く
3. **各操作ステップで CW 4 問サイクルを回す**:

   ```
   ── ステップ N 開始 ──
   
   [Q1 ゴール形成] 操作前
   「私は今から〇〇しようとしている」を一人称で宣言する。
   → 判定: このステップで何をすべきか分かったか？ (Yes/No + 理由)
   
   [Q2 操作可視性] snapshot 確認
   `browser_snapshot` で現在画面を確認する。
   → 判定: やるべき操作が画面上に見えているか？ (Yes/No + 理由)
   → 見えない場合: スクロール or 探す行動をシミュレート
   
   [Q3 操作-効果対応] 判断
   見えている操作のうち、どれをクリック/入力するか決める。
   → 判定: 選んだ操作が自分のゴールに繋がると確信できるか？ (Yes/No + 理由)
   → 迷った場合: 何秒迷ったか、他の候補は何だったかも記録
   
   [行動] 実行
   `browser_click` や `browser_evaluate` で操作を実行する。
   
   [Q4 フィードバック] 操作後
   `browser_snapshot` で操作後の画面を確認する。
   → 判定: ゴールに近づいたことが分かるか？ (Yes/No + 理由)
   → 分からない場合: 「何が起きたのか分からない」「成功？失敗？」等の心情を記録
   
   [心の声]
   このステップ全体での感想を一人称で記録。
   ここで諦めたくなったか？ (Yes/No)
   
   ── ステップ N 終了 ──
   ```

4. シナリオ完了 or 諦め時点で最終評価:
   - 目的を達成できたか / できなかったか
   - 所要ステップ数（計画 vs 実際）
   - **CW 失敗マップ**: 各ステップで Q1〜Q4 のどれが No だったかのサマリ
   - もう一回使いたいか
   - 一番悪かった瞬間 1 つ（どのステップの何が最悪だったか）
   - 一番良かった瞬間 1 つ（あれば）
   - **机上 CW との差異**: 机上で予測した詰まりポイントは実際に起きたか？机上で見逃していた詰まりはあったか？

5. **必ず `mcp__playwright__browser_close`** で閉じてから次のペルソナへ

各ペルソナのウォークスルーは `work/ux-review/<日付>/walkthroughs/persona-<id>.md` に保存。

### Phase 5.5: 中間集計（機械的クロス集計）

主体: スキル本体が直接実行する（Agent 不要）。Phase 4 と Phase 5 の全結果を読み込んで、**解釈なしの事実集計** を行う。

入力: `reviews/persona-*.md` + `walkthroughs/persona-*.md` の JSON 部分

出力: `work/ux-review/<日付>/aggregation.md` に以下のテーブルを生成:

1. **画面 × Q1〜Q4 クロス集計表**: 各画面で Q1〜Q4 それぞれの Fail 数 / 評価者数。加えて effort=struggled の数も別カラムで出す（Pass だが苦労した = 改善候補）
2. **タスク完遂率テーブル**: タスクごとの completed / with_difficulty / abandoned 集計
3. **ペルソナ一覧と would_use_again 集計**

この Phase はエージェントに「数えて」もらう必要はない。スキル本体が JSON から機械的にカウントしてテーブルを書き出すだけ。統合レポートの数値精度を担保するための前処理。

### Phase 6: 統合レポート

主体: コンサル役 Agent を 1 人立ち上げる。

入力（全て読ませる）:
- `input.md`（ユーザーの選択観点と入力）
- `scout/map.json`
- `scout/tasks.json`
- `personas.md`
- **`aggregation.md`（Phase 5.5 の集計済みテーブル — 数値はここから引用。自分で数え直さない）**
- `reviews/persona-*.md`（机上 CW 全件 — 引用とニュアンスの取得用）
- `walkthroughs/persona-*.md`（実操作 CW 全件 — 引用とニュアンスの取得用）

出力先: `work/ux-review/<日付>/report.md`

レポート構成（**この順番で書く**）:

## 設計上の重要原則

### メモリ節約とブラウザ管理
- **Playwright 実操作は必ずシリアル**。Phase 2 と Phase 5 で 1 セッションしか立ち上がらない設計を守る
- Phase 2 終了時 / 各 Phase 5 ペルソナ終了時に**必ず `browser_close`**。これを忘れると次のペルソナがゾンビセッションを引き継いで状態が汚染される
- 並列するのは Agent 側 LLM だけ（Phase 4）。ブラウザは並列にしない

### ペルソナの役柄固定
- ペルソナエージェントには「あなたはレビュアーではない」「お世辞禁止」「専門用語禁止」を毎回明示する
- `voice_signature` で口調を固定し、`mental_model` で事前知識レベルを固定する
- 役を演じきれないと CW が「普通の UX 評論」になってしまい、認知障壁の多様性が失われる
- schema の `raw_voice` と `narration` フィールドは特に「整形しない、口語で、voice_signature の口調で」と強調する

### CW 4 問の品質管理
- 各ステップで 4 問全てに明確な Yes/No + effort と理由を出させる。「どちらとも言えない」は許可しない（強いて言えばどちらか、を決めさせる）
- 理由は一人称で書かせる。「操作が見つけにくい」ではなく「えっ、ボタンどこ？画面の端っこ？こんなの気づかないよ」
- **effort=struggled は Fail と同等に改善候補**。Pass/Fail 二値だけ見ると struggled が埋もれる。Phase 5.5 の集計で struggled 数を別計上する
- tech_literacy が high のペルソナが全ステップ Pass + instant だけで終わると情報量が薄い。高リテラシーでも「ラベルの意味が不明」（Q3 Fail）や「一瞬どっちか迷った」（Q3 struggled）等はありうるので、安易に全 instant Pass にしない旨をプロンプトに含める

### 成果物の分割
- 偵察結果を 1 つの巨大 JSON にしない。画面ごとにファイルを分けて、ペルソナエージェントに渡すときも参照しやすくする
- ペルソナ机上 CW も 1 ファイル/人。後段の統合レポート Agent が並列に読み込めるように

### 中断と再開
- 各 Phase が終わるたびに成果物が `work/ux-review/<日付>/` に残る
- 中断された場合、次回のセッションで「どこまでできているか」を見るだけで再開地点が分かるように
- Phase 6 の統合レポートだけは「やり直したい」要求が出やすいので、コンサル役エージェントを差し替えやすい設計を保つ

### 認証情報の扱い
- ユーザーから受け取った認証情報は**成果物ファイルに書かない**
- 必要なのは Phase 2 偵察と Phase 5 実操作のみ。会話メモリ内で扱う
- ログに残す場合は `***` でマスクする
- テスト用アカウントを推奨する旨を Phase 1 入力収集時に伝える

### スコープ管理
- 偵察画面数は 5〜10 に抑える（深掘りしすぎると Phase 4 の入力が肥大化する）
- タスク分解は 3〜5 タスクに抑える。タスクあたりのステップも 3〜10 が目安
- ペルソナ数は 8〜12 が標準。少なすぎると観点カバレッジが落ち、多すぎると統合レポートの収束が悪い
- 実操作するペルソナは 2〜3 人。これ以上だと時間とブラウザリソースが厳しい

## 既存スキルとの関係

- `frontend-design-review`: 静的なコード/PR レビュー寄り。**目的が違うので共存**。デザインのコードを見たいときはそちらを使う
- `parallel-task-orchestrator`: 並列パターンの設計思想（メインスレッドで分割、サブセッションで実行）が近い。Phase 4 の並列起動の参考になる
- `frontend-design`: 新規 UI デザイン作成スキル。CW の改善提案を受けて作り直すときに連携してもよい

## 失敗時のフォールバック

- **Playwright MCP が使えない / 許可されていない**: Phase 2 をスキップしてユーザーに「主要画面のスクショを添付してください」と依頼する。タスク分解はユーザーとの対話で作る。スクショだけで Phase 3〜4 は実行可能（Phase 5 は不可なので机上 CW のみで終わる）
- **ペルソナエージェントが CW 4 問を形式的にこなす**（全部 Pass にしてしまう）: プロンプトに「Q1〜Q4 のうち少なくとも 1 つは No か struggled になるステップがあるはず。全 instant Pass は疑ってかかれ。あなたの tech_literacy と mental_model を考慮して本当に一瞬で迷わなかったか？」と追加する
- **ペルソナエージェントが役を演じきれない**（"プロのレビュアー口調になっちゃう"）: プロンプトの先頭に「あなたは絶対に〇〇です。レビュアーの言葉遣いをしたら失格です」と更に強める。それでも崩れる場合は schema の `raw_voice` を必須にして検閲する
- **偵察画面が多すぎて入力肥大**: 構造 JSON から「テキストの長文」を削って要素名と階層だけ残す。スクショだけ参照させる方針に切り替える
- **実操作で詰まる**（フォーム入力ができない等）: `browser_evaluate` で JS を直接実行して状態を進める。それでも無理ならそのペルソナのウォークスルーは「ここで詰んだ」と記録して打ち切る（これ自体が CW の貴重な Fail データ）
- **タスク分解が実態と合わない**（偵察時と実操作時で画面が違う等）: 実操作 CW では柔軟にステップを追加・変更してよい。ただし逸脱した箇所を明記し、統合レポートの「机上 vs 実操作の差異分析」に反映する
