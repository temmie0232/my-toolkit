# Anki Card Generator

教材を読み込み、AnkiにインポートできるTSVファイルを生成する。

## 設計原則

**1. recall優先。** 自力で思い出す練習（recall）は、4択本番のような再認形式のテストにも転移する。
逆（選択肢を選ぶ練習→想起力）は成り立ちにくい。生成カードは原則すべてrecall（Basic / Cloze）にする。
**MCQを使うのは2箇所だけ**: ①模擬試験・過去問の逐語変換（本番の出題形式と誤選択肢の文面を
保存する価値がある）、②答えを一意にできないシナリオ問い。それ以外をMCQにしない —
静的カードでは正解の選択肢位置を数回で覚えてしまい、毎回選択肢を4つ読むためレビューも遅い
（recall 5〜10秒/枚 vs MCQ 20〜40秒/枚）。本番形式への慣れは模試逐語カードが担う分業とする。

**2. 答えの原子性。** recallカードの裏面は一語〜1行で、正誤を客観判定できる形にする。
散文の再現を求めない（自己採点がブレると、スケジューラの間隔計算もブレる）。
定義カードは「分かれ目キーワード」を太字にし、**太字を言えたら正解**という採点基準をカード自体が示す。

**3. 復習時間は有限資源。** 生成カードの価値は「試験に出る × 手持ちカードで触れる機会が少ない」で
決まる。頻出でも模試カードが厚くカバーするトピックには積まない。迷ったら作らない。

| 知識タイプ | 形式 | ノートタイプ |
|---|---|---|
| 用語 → 定義 | recall Basic（裏面=分かれ目キーワード太字の1行） | 基本 |
| 定義 → 用語（逆引き） | recall Basic（答え一語。最も試験転移が高い） | 基本 |
| 数値・上限・期間・列挙・固有事実 | Cloze | 穴埋め |
| 混同ペアの区別 | recall Basic「AとBの分かれ目は？」 | 基本 |
| シナリオ → 概念・サービス | recall Basic（答えが一意になるよう書く） | 基本 |
| シナリオで答えを一意にできない場合のみ | MCQ | 基本 |
| 模擬試験・過去問 | 逐語MCQ | 基本 |
| コード読解（出力予測等） | 逐語MCQ（エスケープ必須） | 基本 |
| 対訳ドリル | Basic（出題→答え） | 基本 |
| 単語・用語メモ | recall Basic（双方向）/ Cloze | 基本/穴埋め |

## 教材フォーマット

共通フォーマット規約準拠の教材を前提とする。
規約: `~/.claude/skills/shared/references/study-material-format.md`

規約外の教材でも、見出し階層やテーブル構造から推測して対応する。
大型教材は `sessions/index.md` を入口とする分割形式のことがある（index.md の目次から辿る）。

## 起動

- `/anki` -- 教材一覧 → 選択 → カード生成
- `/anki <パス1> <パス2> ...` -- 指定ファイルから直接生成

## 手順

### Step 1: 教材の選択

`~/learning/` 配下を走査し、Markdownファイルを一覧表示する。

```
カード生成対象を選んでください（番号で指定、複数可）:

AZ900/
  1. AZ900.md（参考書）
  2. AZ900_模擬試験1.md
  ...

ENG/瞬間英作文/
  7. 瞬間英作文.md
```

番号または名前で選択させる。複数選択可（例: 「2, 3, 4」）。

### Step 2: 教材ジャンルの判別

ファイルの内容からジャンルを判別し、生成するカードの構成を決める:

| ジャンル | 判別基準 | 生成するカード |
|---------|---------|--------------|
| 参考書型 | Must Memorize表・解説文が主体 | recall Basic / Cloze 中心（→「概念系の生成」） |
| 問題集型 | 過去問チャレンジが主体、コード付き問題 | 逐語MCQ ＋ Must Memorize表からrecall/Cloze |
| 模擬試験・過去問 | 問題ブロックの連続 | 逐語MCQ |
| ドリル型 | ドリル表（答え列見出しに「答え」） | Basic（出題→答え） |
| 単語帳 | 語彙の対応ペア | Basic双方向（1語2枚） |
| メモ系 | 規約外。「用語: 定義」の短行羅列 | recall Basic双方向 ＋ 数値行はCloze |
| 論述・記述式対策 | 減点パターン表・設問の型・一問一答 | recall Basic / Cloze（答案演習はカード化しない） |
| 読み物ダイジェスト | 一枚まとめ・キーコンセプト表・章ダイジェスト構成 | 原則対象外。ユーザーが明示的に望む場合のみキーコンセプト表から recall Basic / Cloze |

判別結果と生成方針をユーザーに確認する
（例:「AZ900.md → 参考書型。recall Basic/Cloze中心、模試逐語分と重複するトピックは薄めに生成します。OK？」）。

### Step 3: セクション単位で段階的にカード生成

教材が大きい場合（複数セッションがある場合）、セクション（`#` 見出し）ごとに処理する:

1. ファイル全体の見出し構造を先に把握
2. 1セクションずつ読み込み → カード生成 → 蓄積
3. 進捗を報告する（「Session 3 のカード生成中... (3/10)」）

### Step 4: TSV出力

全カードを1つのTSVファイルにまとめて出力する。

- 出力先: `~/learning/anki/`（なければ作成）
- ファイル名: `<教材ディレクトリ名>_<YYYY-MM-DD>.tsv`
- 複数ディレクトリの教材を混ぜた場合: `<dir1>_<dir2>_<YYYY-MM-DD>.tsv`

### Step 5: 推奨デッキオプションの提示（毎回・全項目）

カード生成後、割り当てるデッキの **デッキオプション推奨を毎回・全38項目** 提示する。
仕様の正典: `references/deck-presets.md`（2プリセット・写経テンプレート・逆算式・運用カレンダーの唯一の正)。
**実行時は必ず deck-presets.md を読み直す**（記憶で再構成しない）。

分ける基準は「カードの種類」ではなく **締め切りの有無**（FSRS が1枚ごとの難易度を吸収するので、
人間がプリセットで決めるのは締切まわりの方針＝`復習間隔の上限` と `目標正答率` だけ）。

1. 生成した **新規カードの枚数** を数える。既存デッキへの追加なら、デッキに残っている
   未導入分も合算する（枚数が分からなければユーザーに聞く）
2. **確認質問を1ターンに束ねて聞く**: 締切の有無（文脈から試験日の候補が分かるときは
   「◯◯試験 YYYY-MM-DD でよいですか？」とYes/No形式で・なければ日付を質問。聞かずに決めない）
   ＋（任意）1日にAnkiへ使える時間・負担を軽くしたい曜日
   - 締切なし → **プリセットA（一生モノ）** ／ 締切あり → **プリセットB（締切あり）**
   - **締切の有無はジャンルより優先**（単語帳でも試験が近ければ B）
3. `deck-presets.md` の **Part 2**（逆算式とガード）で可変値を計算する
4. **Part 1 のテンプレートを逐語転載**して出力する — サマリ層＋38項目本表＋検算行。
   行の追加・削除・並べ替え・項目名の改名・省略記号を禁止。**全38行を毎回出す**
5. B なら **Part 3 の運用カレンダー**（行文言は固定・日付のみ計算）を続けて出力する
6. 「選択肢の意味も知りたい」と言われたら Part 1 の **対照表**（全選択肢＋意味）を提示する

## TSVフォーマット

ヘッダ付き・タブ区切り・4カラム（Anki 2.1.54+ のファイルヘッダでノートタイプ混在を実現する）。

```
#separator:tab
#html:true
#notetype column:1
#tags column:4
基本	<表面>	<裏面>	<タグ>
穴埋め	<{{c1::〜}}を含む本文>	<補足（なければ空）>	<タグ>
```

- 使うノートタイプは「基本」「穴埋め」の**2つに限定**する（インポート事故を減らす。
  双方向カードは明示的に2行生成し、「裏表反転付き」ノートタイプは使わない）
- ノートタイプ名はユーザーのAnkiに存在する名前と一致させる必要がある。
  日本語版Ankiの標準名が「基本」「穴埋め」。英語版なら Basic / Cloze に置換するよう案内する
- フィールド内の改行は `<br>`（HTMLタグ）。太字は `<b>`、その他HTMLタグで整形可
- フィールド内にリテラルなタブ・改行を入れない
- タグは**スペース区切り**で複数指定

### タグ構成

```
<教材名>::<セッション>::<トピック> <教材名>::source::<種別>
```

**内容階層タグ** — カードの分類:
```
AZ900::Session1::クラウドの基本概念
AZ900::Session6::ID管理
瞬間英作文::Session1::be動詞
```

**ソース種別タグ** — 出典:
- `source::textbook` — 参考書の本文・表から生成
- `source::mock-exam` — 模擬試験をそのまま変換
- `source::generated` — 教材内容をもとに新規作成した問題

過去問チャレンジの問題は、見出しの出典表記で source タグを決める（規約で必須化済み）:
`（出典: 模擬試験N 問M）` → mock-exam ／ `（出典: 原本 <章>）` → textbook ／
`（出典: オリジナル）` → generated。出典表記が無い古い教材では文脈から推測する。

模擬試験ファイルから直接カード化するとき、教材ディレクトリに `analysis.md` の
「模擬試験 → セッション対応表」があれば、内容階層タグを `<教材名>::SessionN::<トピック>`
として付ける（対応表が無ければ `<教材名>::模擬試験N` 止まりでよい）。

## 枚数の重み付け: 頻出度 × 模試カバレッジ

1概念あたり一律ではなく、**0〜5枚**に振り分ける。判定は2軸:

1. **頻出度** — `analysis.md` の「トピック別出題頻度」。なければ Must Memorize表掲載・
   `**混同注意**:` の有無を代理指標にする
2. **模試カバレッジ** — 「模擬試験 → セッション対応表」から、そのトピックが模試逐語カード
   （生成済み・生成予定）で何問カバーされるかを数える

| 頻出度 | 模試カバレッジ | 枚数と構成 |
|--------|--------------|-----------|
| 高 | 薄い（〜2問） | 4〜5枚: 逆引き＋定義＋分かれ目＋Cloze＋シナリオ |
| 高 | 厚い（3問〜） | 1〜2枚: 逆引き・Cloze中心。模試カードが問わない角度だけ補う |
| 中 | — | 2〜3枚: 逆引き＋定義（＋数値があればCloze） |
| 低 | — | 0〜1枚: 逆引きのみ、または作らない |

- analysis.md がない教材では頻出度軸のみで判定する（高=混同注意付き・Must Memorize掲載、低=本文のみ）
- 迷ったら少なめに倒す。カードは復習時間という有限資源を消費する。
  総枚数の多さより、毎日回し切れることが優先

## カード生成: 概念系（参考書型）

### recall Basic: 逆引き問い（定義 → 用語）

最優先のカード。答えが一語で自己採点が明確、かつ「説明文→どの概念か」は試験の出題形式
そのものなので転移が最も高い。ソース: Must Memorize表の各行。

```
表面: 「需要に応じて自動的にリソースを増減する能力」— この用語は？
裏面: <b>弾力性（Elasticity）</b>
```

### recall Basic: 定義問い（用語 → 定義）

表面は用語だけを問い、選択肢を付けない。**裏面は分かれ目キーワードを太字にした1行**に収める。
採点基準は「太字キーワードを言えたら正解」— 定義文の一字一句の再現は求めない。

```
表面: <b>弾力性（Elasticity）とは？</b>
裏面: 需要に応じて<b>自動的に</b>リソースを増減する能力<br>
      <small>採点: 「自動」が言えたらOK／スケーラビリティ（手動含む）との分かれ目</small>
```

### Cloze: 数値・固有事実・列挙

ソース: `## 数値まとめ` 表と本文中の数値・上限・期間・SLA・固有名。

```
穴埋め	可用性セットの障害ドメインは最大{{c1::3}}		AZ900::Session4::可用性 AZ900::source::textbook
穴埋め	VMのSLAは可用性ゾーン構成で{{c1::99.99%}}、可用性セットで{{c2::99.95%}}		AZ900::Session4::可用性 AZ900::source::textbook
```

- 1行に複数の `{{cN::}}` を置くと1ノートから複数カードが生成される。
  関連する数値はまとめてよい（1ノート3個まで目安）
- 列挙（「〜の3要素」等）もClozeで: 各要素を別の `cN` にする
- 補足フィールド（3カラム目）は空でもよい。覚えるヒントがあれば入れる

### recall Basic: 分かれ目問い（混同ペアの区別）

ソース: `**混同注意**:` マーカーと比較表。選択肢は付けない — 識別の核心である
「分かれ目」そのものを想起させる。

```
表面: <b>弾力性とスケーラビリティの分かれ目は？</b>
裏面: <b>自動かどうか</b><br>
      <small>弾力性=需要に応じて自動増減／スケーラビリティ=手動を含む拡張能力</small>
```

### recall Basic: シナリオ問い

具体的な場面から概念・サービスを想起させる。**答えが一意になるようにシナリオを書く**
（教材の具体例・試験Tipsをソースにする）。

```
表面: ECサイトで、セール時にアクセスが急増し、終了後は元に戻る。<br>この需要変動に対応する特性は？
裏面: <b>弾力性（Elasticity）</b>
```

どうしても答えを一意にできない場合（複数サービスが正解になりうる等）のみ、
4択MCQにする（→「MCQのルール」）。その場合もシナリオを絞って一意化できないか先に試すこと。

## カード生成: 模擬試験・過去問チャレンジ（逐語変換）

模擬試験・過去問チャレンジの問題は、本番形式を保存するため**そのままMCQカードに変換**する。
本番の出題形式・誤選択肢の文面に触れる役割はこのカード群が担う。

- 表面: 問題文 + 選択肢（原文のまま）
- 裏面: 正解 + 解説（原文のまま）+ 各誤選択肢の理由

**逐語MCQ変換の対象は実試験由来（`source::mock-exam` / `source::textbook`）のみ。**

**一問一答形式の問題**（`**答え: ...**` マーカー。出典オリジナル・理解確認・論述対策等）は
recall Basic に変換する: 表=問題文、裏=答え（＋補足解説があれば `<small>` で）。

**旧教材の出典オリジナル4択**（一問一答化前の教材に残っているもの）は逐語変換しない。
問われている知識を抽出し、recall Basic（逆引き・分かれ目等）または Cloze に変換する
（生成MCQを作らない設計を維持するため）。

**はい/いいえ形式の問題:** 2択の問題でも、MCQ形式（4択）に変換する。
元の2択を含めつつ、関連する誤選択肢を2つ追加して4択にする。

**重複排除:**
- 問題文と選択肢が実質同一の問題 → 1つだけ残す
- 同じ概念を問うが問い方・選択肢が異なる → 両方残す（異なる角度として有効）

### MCQのルール（逐語変換・例外シナリオ共通）

裏面は、正解の説明に加えて、**全ての誤選択肢について「なぜ間違いか」を1行ずつ必ず記載する**。
省略しない。正解の位置はカード間でばらけさせる。

```
<b>正解: X</b><br><br>
正解の説明文。<br><br>
<b>他の選択肢:</b><br>
A: なぜAが間違いか（1行で簡潔に）<br>
B: なぜBが間違いか<br>
D: なぜDが間違いか
```

**ディストラクターの原則**（はい/いいえ→4択化や例外シナリオMCQで選択肢を新規に作るとき）:
- **同じセクション・同じカテゴリ**の概念から選ぶ。無関係な選択肢は使わない
- **もっともらしいが明確に誤り**であること
- 教材内で「紛らわしい」「混同注意」と示されている概念ペアを優先的に使う
- 教材に載っていない概念を混ぜない

## カード生成: 問題集型（コード付き問題）

原本の問題は模擬試験と同様に逐語MCQ変換する。加えて Must Memorize表・ひっかけパターン表
からは recall Basic / Cloze も生成する（重み付けルールに従う）。

**コードのエスケープ（必須）:** コードを含むフィールドは次の手順で変換する。

1. `&` → `&amp;`、`<` → `&lt;`、`>` → `&gt;` の順で置換
2. 改行 → `<br>`
3. 全体を `<pre><code>` 〜 `</code></pre>` で包む（インデントの半角スペースは保持される）

```
表面: 次のコードをコンパイル・実行するとどうなる？<br>
<pre><code>int[] arr = {1, 2, 3};<br>System.out.println(arr.length);</code></pre><br>
A. 3<br>B. 2<br>C. コンパイルエラー<br>D. 実行時例外
```

## カード生成: ドリル/単語系（Basic）

### 対訳ドリル

ドリル表の各行から1枚のBasicカードを生成する。表=出題列、裏=答え列。

```
表面: これはあなたのノートですか。
裏面: Is this your notebook?
```

文法ノートや注意点がある場合は、裏面の答えの下に補足として追加する:

```
裏面: Is this your notebook?<br><br><small>be動詞の疑問文: be動詞を主語の前に</small>
```

### 単語帳

1語につき**双方向2枚**を生成する（語→意味、意味→語）。例文・注意があれば裏面に
`<small>` で補足する。双方向が不要な教材（産出方向だけ鍛えたい等）はユーザーに確認する。

### メモ系

「用語: 定義」の短行羅列（学習メモ等）から、逆引き＋定義問いのrecall Basicを生成する。
数値・固有値を含む行はClozeにする。

## 生成後の報告

カード生成完了時に以下を表示する:

1. **生成枚数**（recall Basic / Cloze / MCQ の内訳。MCQはさらに逐語変換/例外シナリオを区別）
2. **出力ファイルパス**
3. **タグ一覧**（ユニークなタグをリスト表示）
4. **推奨デッキ設定**（Step 5＝サマリ＋全38項目の写経出力＋Bなら運用カレンダー。正典は `references/deck-presets.md`）
5. **インポート手順**:

```
Ankiへのインポート:
1. Anki → ファイル → インポート → 生成されたTSVファイルを選択
2. ファイル先頭のヘッダにより、セパレータ・HTML・ノートタイプ列・タグ列は自動設定される
3. 「ノートタイプが見つからない」エラーが出る場合は、TSV冒頭のノートタイプ名
   （基本 / 穴埋め）を自分のAnkiのノートタイプ名（英語版: Basic / Cloze）に置換して再実行
4. インポート実行
```

## ルール

- 教材の事実を改変しない。カード化のための言い換え・再構成はOKだが、内容の正確性を維持する
- 模擬試験の正解・解説は原文を正確に保持する
- 生成カードはrecall（Basic/Cloze）。MCQにするのは試験問題の逐語変換と、
  答えを一意にできないシナリオ問いのみ
- recallカードの裏面は一語〜1行で客観採点できる形にする。散文の再現を求めない
- ディストラクターは教材内の概念のみ使う。教材にない情報を混ぜない
- コードを含むフィールドは必ずエスケープ規定に従う
- 出力先 `~/learning/anki/` がなければ作成する
- 大きな教材はセクション単位で処理し、進捗を都度報告する
- カード生成後は必ず推奨デッキ設定（Step 5）を提示する。締切の有無を確認してから
  プリセットA/Bを決める（聞かずに勝手に決めない）。出力は `references/deck-presets.md` の
  写経テンプレートを逐語転載する（サマリ＋38項目＋検算行。省略・並べ替え・改名・言い換え禁止）
